# DailyForex Premium — AI Trading Signal Engine (v2.3.0)

A production FastAPI trading signal engine for forex, crypto, commodities, and indices. Signals are generated by five QuantConnect-validated strategies, persisted to SQLite, and served via a read-only REST API with WebSocket real-time broadcast.

---

## Stack

| Layer | Technology |
|---|---|
| API | FastAPI + Uvicorn |
| Scheduler | APScheduler (cron, background) |
| Database | SQLite + SQLAlchemy (WAL mode) |
| Cache | 4-shard TTLCache (60s default) |
| Market Data | FCSAPI v4 |
| Notifications | Discord / Slack / Generic webhook |
| Publishing | WordPress REST API (Application Passwords) |
| Real-time | WebSocket (`/ws/signals`) |
| Startup | start.sh (run the FastAPI engine) |
| Backfill Tool | backfill.py (standalone CLI tool) |
| Restore Tools | restore_signal.py, restore_closed_signals.py, restore_trend_non_forex.py |

## Data Synchronization

The engine implements a three-layer FCSAPI data sync system to ensure candle data is always fresh and entry prices are validated before signals are generated. This prevents inaccurate signals caused by stale candles, missed scheduler runs, and unvalidated entry prices.

### Layer 1 — Pre-evaluation Candle Freshness Check
- Automatic staleness detection for cached candles
- Timeframe-specific thresholds: D1 (24h), 4H (5h), 1H (2h), 30m (1h)
- Automatic force-refresh when data is stale
- Logging of freshness status and refresh actions

### Layer 2 — Entry Price Validation
- Validates entry prices against current market data before signal generation
- Ensures signals are based on accurate, up-to-date prices
- Prevents signals with outdated or incorrect entry levels

### Layer 3 — Startup Candle Sync
- Ensures all assets have fresh data on engine startup
- Comprehensive sync across all active strategies and timeframes
- Prevents stale data issues after server restarts or deployments

## Signal Management

Signals are read-only via the public `/api/v1/` endpoints. Closing
or removing a signal requires admin authentication and must be done
via the admin API or the maintenance tools.

### Close a Signal via Admin API

**Force-close by strategy and asset (recommended):**
```http
POST /admin/api/force-close-position
Content-Type: application/json

{
  "strategy_name": "mtf_ema",
  "asset": "BTC/USD",
  "exit_price": 70500.00,
  "exit_reason": "Manual admin close"
}
```

Response:
```json
{
  "status": "closed",
  "position_id": 12,
  "strategy_name": "mtf_ema",
  "asset": "BTC/USD",
  "exit_price": 70500.00,
  "exit_reason": "Manual admin close",
  "signals_closed": [45]
}
```

**Close via browser console (while logged into admin panel):**
```javascript
fetch('/admin/api/force-close-position', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    strategy_name: 'mtf_ema',
    asset: 'BTC/USD',
    exit_price: 70500.00,
    exit_reason: 'Manual close'
  })
}).then(r => r.json()).then(console.log)
```

### Restore a Missing Signal

When a live trade exists in your broker account but is missing from
the engine DB (caused by server sleep mid-transaction or DB reset),
use `restore_signal.py` to re-insert it:
```bash
# Dry run first
python3 restore_signal.py \
  --asset "BTC/USD" \
  --direction SELL \
  --entry 70960.65 \
  --stop-loss 71967.22 \
  --take-profit 67121.657 \
  --strategy mtf_ema \
  --timestamp "2026-03-25T12:00:00" \
  --dry-run

# Actual restore
python3 restore_signal.py \
  --asset "BTC/USD" \
  --direction SELL \
  --entry 70960.65 \
  --stop-loss 71967.22 \
  --take-profit 67121.657 \
  --strategy mtf_ema \
  --timestamp "2026-03-25T12:00:00"
```

ATR at entry is auto-derived from entry/stop-loss if not provided
using each strategy's SL multiplier.

## Security

- Multi-layer leaky bucket rate limiting: 20 req/2s burst · 60/min · 1,000/hr per IP
- Burst cooldown: 5 minutes
- Endpoint enumeration guard: 5+ 404s in 60s → 24h IP block
- Partner API keys with tier-based limits (Standard / Premium / Unlimited)
- CORS restricted to `*.dailyforex.com` + configured origins — wildcard blocked
- Security headers: HSTS, X-Frame-Options, X-Content-Type-Options, CSP, Referrer-Policy
- 1 MB payload limit on all POST/PUT/PATCH endpoints
- Session-based admin auth (PBKDF2-SHA256, 24h expiry)

## Maintenance Tools

### Backfill Missing Candle Data
When the server has been sleeping (Replit free tier) and candle data
is missing, run the backfill tool to re-fetch up to 300 candles per
asset/timeframe from FCSAPI:
```bash
# Dry run first to check credit cost
python3 backfill.py --dry-run

# Refresh everything
python3 backfill.py --force

# Refresh a single asset
python3 backfill.py --symbol "XAU/USD" --force

# Refresh a single strategy
python3 backfill.py --strategy mtf_ema --force

# Refresh a single timeframe across all strategies
python3 backfill.py --timeframe D1 --force
```

### Restore Closed Signals
Use `restore_closed_signals.py` to insert specific closed signals into the
staging or live database so the stored history matches the live site.
```bash
# Insert predefined closed signals into staging DB
python3 restore_closed_signals.py

# Preview only, no writes
python3 restore_closed_signals.py --dry-run

# Force skip confirmation
python3 restore_closed_signals.py --force

# Target the live database explicitly
python3 restore_closed_signals.py --db-path trading_engine/trading_data.db
```

### Restore Trend Non-Forex Assets
Use `restore_trend_non_forex.py` to verify and insert the configured
trend_non_forex strategy asset symbols after FCSAPI validation:
```bash
# Verify and insert all trend_non_forex symbols
python3 restore_trend_non_forex.py

# Verify only, do not write to the DB
python3 restore_trend_non_forex.py --dry-run

# Skip the credit confirmation prompt
python3 restore_trend_non_forex.py --force
```

### [2.1.3] — 2026-03-25

#### MTF EMA — QC Algo Alignment (Entry Logic)

**`multi_timeframe.py`**
- Fixed Condition 1 (C1) LONG entry: now requires `price > D1 EMA50 AND price > D1 EMA200` in addition to `EMA50 > EMA200`. Previously only the EMA cross was checked — price location was ignored, allowing LONG entries during deep pullbacks where price was below both EMAs.
- Fixed Condition 1 (C1) SHORT entry: now requires `price < D1 EMA50 AND price < D1 EMA200` in addition to `EMA50 < EMA200`. Prevents SHORT entries firing at the crossover candle while price is still elevated above both EMAs.
- Added `LONG_ONLY_ASSETS = {SPX, NDX, RUT, DJI, OSX}` constant — indices and equity-like instruments are restricted to LONG signals only, matching QC algo behavior where `SecurityType.EQUITY` and `SecurityType.FUTURE` cannot go short.
- Added `_check_long_only_entry()` method — dedicated LONG-only entry path for restricted assets, routed from `evaluate()`.

**Root cause context**

The engine uses EMA50/EMA200 on D1 (not SMA50/SMA200). EMA reacts faster to price moves than SMA, so during trend transitions the two indicators can give conflicting readings. An integrity check on the GBP/USD SHORT signal confirmed the signal was valid (EMA death cross occurred Mar 18, price was below both EMAs at entry), but the audit exposed the missing price location gate as a latent bug that could cause premature entries on future signals.
