TASK 1 — SIGNAL DEDUPLICATION (Backend + Frontend, all strategies)
=============================================================

PROBLEM:
- Duplicate signals appear for the same asset+strategy+direction because multiple evaluations fire before the DB unique constraint catches them, or because `get_all_signals()` returns multiple CLOSED rows for the same asset+strategy pair.
- The frontend Active Signals panel and the backend Trading Signals tab both show duplicates.

RULES:
1. One OPEN signal per (strategy_name + asset) at any time. No exceptions.
2. For CLOSED signals displayed in any list or API response, show only the single most recent closed signal per (strategy_name + asset + direction) — determined by the highest `id` (i.e. latest insert).
3. The "latest timestamp" rule: when deduplicating, always keep the record with the highest `signal_timestamp`. If two records share the same timestamp, keep the one with the highest `id`.
4. Apply this rule in every place signals are read and returned — backend DB query, admin panel HTML table, and all API endpoints.

TASK 2 — PREVIOUS SIGNALS: 3-MONTH RETENTION LIMIT
=============================================================

PROBLEM:
- The database grows unboundedly as closed signals accumulate.
- The frontend Previous Signals panel (and history API) currently returns all closed signals with no age limit.
- We want to cap displayed/returned data at 3 months (92 days) AND purge older closed signals from the database on a schedule.

RULES:
1. Display limit: Previous Signals on the frontend and `/api/v1/signals/history` must never return CLOSED signals older than 92 days. OPEN signals are always returned regardless of age.
2. Database purge: A scheduled job runs daily at 2:00 AM ET and hard-deletes CLOSED signals older than 92 days from the `signals` table. Log the count of deleted rows.
3. The purge must also clean up orphaned rows in `signal_cms_posts` for any deleted signal IDs (cascade or manual delete).
4. Signal metrics (`signal_metrics` table) are NOT affected — metrics are aggregated and do not store raw signal rows.
5. The 92-day window is defined as: signal_timestamp < (NOW UTC - 92 days). Use the `signal_timestamp` column, not `created_at`.

CHANGES REQUIRED:

A) database.py — modify `get_all_signals()`
- Add an optional parameter: `max_age_days: Optional[int] = None`
- When `max_age_days` is set, add a WHERE clause: `signal_timestamp >= datetime('now', '-{max_age_days} days')` for CLOSED signals only. OPEN signals are always included.
- Do NOT change the function signature in a way that breaks existing callers — default is None (no filter, existing behavior preserved).

B) database.py — add new function `purge_old_closed_signals(days: int = 92) -> dict`
- Hard-deletes CLOSED signals where signal_timestamp < (NOW - days).
- Before deleting signals, collect their IDs and delete matching rows from `signal_cms_posts`.
- Returns a dict: `{"deleted_signals": N, "deleted_cms_posts": M, "cutoff_date": "YYYY-MM-DD"}`.
- Wrap in try/except, rollback on error, log the result at INFO level.
- Add a DB index on `signals(status, signal_timestamp)` if it does not already exist (check models.py — add to `__table_args__` if missing).

C) main.py — add scheduled purge job
- Add a new APScheduler job that calls `purge_old_closed_signals(days=92)` daily at 2:00 AM ET.

- Register this job alongside the existing jobs in the `lifespan()` function.

D) api_v1.py — `get_signals_history()`
- Pass `max_age_days=92` when calling `get_all_signals()` so the history endpoint never returns signals older than 92 days.
- Keep OPEN signals unfiltered (the DB function handles this).
